操作系统接口

进程与内存

fork()用于通过系统调用创建子进程，其在父子进程中都有返回值。在父进程中返回子进程的pid；在子进程中返回0，因此，可以根据返回值判断父子进程。

int pid = fork();
if(pid > 0) {  // 父进程
    printf("parent: child=%d\n", pid);
    pid = wait((int *) 0);
    printf("child %d is done\n", pid);
} else if(pid == 0) {  // 子进程
    printf("child: exiting\n");
    exit(0);
} else {  // error
    printf("fork error\n");
}

exit系统调用用于进程停止执行并释放资源，其接收一个整数状态参数，通常0表示成功，1表示失败。

wait系统调用返回当前进程已退出（或已杀死）的子进程pid，并将子进程的退出状态传递给wait地址；如果子进程没有退出，则wait等待一个子进程退出。如果调用者没有子进程，wait立即返回-1。如果父进程不关心子进程退出状态，可以传递0地址给wait。

父子进程拥有相同的内存内容，但是二者运行在不同的内存空间中，因此，变量不会相互影响。

IO和文件描述符

read和write都是以字节为单位读取指定fd（文件描述符）中的内容的。

read(fd, buf, n)表示从fd中读取最多n个字节，将它们复制到buf中并返回读取的字节数。read每次从文件偏移量读取数据，即后续会读取之前读取之后的字节。没有字节可读时，read返回0表示文件结束。

write(fd, buf, n)将buf中的n个字节写入fd，并返回写入字节数，发生错误时才会写入小于n字节的数据。其余与read类似。

dup复制现有的文件描述符，返回一个引用自同一个底层对象的新fd。两个fd共享一个偏移量。

fd = dup(1);
write(1, "hello ", 6);
write(fd, "world\n", 6);

管道是一对文件描述符提供给进程的小型内核缓冲区，为进程提供了一种通信方式。

int p[2];

pipe(p);
if (fork() == 0) {
    close(0);
    dup(p[0]);
    close(p[0]);
    close(p[1]);
    exec("/bin/wc", argv);
} else {
    close(p[0]);
    write(p[1], "hello world\n", 12);
    close(p[1]);
}

pipe，创建管道，在数组p中记录读写的fd。子进程调用close和dup使得fd 0指向读取端。关闭p中所存的fd，调用exec运行wc。

close是释放一个文件描述符，使其未来可被open、pipe或dup调用。新分配的文件描述符总是当前进程中编号最小的未使用描述符。

TIPS：0对应的是标准输入， 1对应标准输出， 2对应标准错误输出。

父进程关闭管道读取端，写入管道，然后关闭写入端。

在没有数据进来的时候，管道的写端会进入等待，直到有数据写入或所有指向写入端的fd都被关闭。

操作系统架构

操作系统必须满足的三个要求：多路复用、隔离和交互。

抽象系统资源

隔离可以避免应用程序直接访问硬件资源，将资源抽象为服务。例如，可以通过open、read、write和close与系统交互，而不是直接读取磁盘。同时，这在相互竞争的状态下更高效获取资源。

Unix的进程交互通过文件描述符实现，不仅抽象了细节（比如数据的具体存放位置），还简化了交互方式的定义。

用户态、核心态以及系统调用

强隔离需要应用程序与操作系统直接存在硬边界，应用程序出错不应影响其他程序或操作系统，操作系统必须保证应用程序不能修改或读取操作系统的数据结构和指令，不饿能访问其他进程的内存。

CPU为强隔离提供了硬件支持。例如，RISC-V的三种执行指令的模式：机器模式，管理模式和用户模式。

用户想要调用内核应用程序必须过渡到内核态。由内核控制转换奥管理模式的入口点，否则，恶意程序可以跳过参数验证的地方进入内核。

内核组织

宏内核是整个操作系统都在内核中，不必考虑操作系统那一部分不需要硬件特权，操作系统的不同部分容易合作。但是，操作系统的不同部分接口复杂；内核失败，则所有程序都会失败需要重启。

微内核是在用户模式下执行大部分操作，组织简单，操作系统设计者可以最大限度降低管理模式下运行操作系统的代码量。

进程概述

Unix操作系统的隔离单位是进程。内核用来实现进程的机制包括用户/管理模式编制，地址空间和线程的时间切片。

进程为程序提供了一个看起来像是私有内存系统（其他进程无法读取）和独占的CPU。

Xv6中使用页表为进程提供地址空间。RISC-V页表将虚拟地址映射到物理地址。

xv6为每个进程维护了单独的页表，定义了该进程的地址空间。如图2.3所示，从虚拟地址0开始，从下到上依次是，指令，全局变量，栈区和堆区。在地址空间的顶部，xv6为trampoline（用于在用户和内核之间切换）和映射进程切换到内核的trapframe分别保留了一个页面，第四章会介绍。

xv6内核为每个进程维护了各自的状态并将其聚集到一个proc结构体中，最重要的三个部分是页表，内核栈区和运行状态。

线程用于执行进程指令，其状态存储在线程的栈区上。每个进程有两个栈区：用户栈区和内核栈区。内核栈是独立的，不受用户代码的保护，即使进程破坏了其用户栈，内核仍然可以正常运行。

进程通过执行RISC-V的ecall指令提升硬件特权级别，并将程序计数器（PC）更改为内核定义的入口点，切换到内核态。通过sret指令返回用户空间，降低硬件特权级别，恢复执行用户指令。

编译运行kernel

xv6的整个代码结构由kernel、mkfs和user三个部分组成。

  • kernel中包含了基本所有的内核文件。因为xv6是一个宏内核结构，所有文件会被编译成一个叫做kernel的二进制文件，然后被运行在kernel mode中。
  • user是运行在user mode的程序。
  • mkfs会创建一个空的文件镜像并存放在磁盘上，这样就可以直接使用一个空的文件系统。

接下来，简单介绍内核的编译过程：

首先，Makefile会读取一个C文件，例如proc.c；之后调用gcc编译器，生成一个文件proc.s，这是RISC-V汇编语言文件；再走到汇编解释器，生成proc.o，这事汇编语言的二进制格式。

Makefile会为所有内核文件做同样操作；内核加载器（Loader）会收集所有.o文件，链接生成内核文件。

这里的内核文件就是将在QEMU中运行的文件。

QEMU

QEMU在利用C程序仿真RISC-V主板，但通过QEMU运行内核时候，可以认为运行在真实的RISC-V主板上。

QEMU是一个大型的开源C程序，在其内部按以下流程循环：

  • 读取4字节或8字节的RISC-V指令。
  • 解析RISC-V指令，找出对应操作码
  • 在软件中执行相应指令

为了完成以上操作，QEMU的主循环需要维护寄存器状态，所以QEMU会有以C语言声明的类似X0、X1寄存器等。

传给QEMU的几个参数：

  • -kernel：这里传递的是内核文件（kernel目录下的kernel文件），这是将在QEMU中运行的程序文件。
  • -m：这里传递的是RISC-V虚拟机将会使用的内存数量
  • -smp：这里传递的是虚拟机可以使用的CPU核数
  • -drive：传递的是虚拟机使用的磁盘驱动，这里传入的是fs.img文件

页表

页式硬件

页表是在硬件中通过处理器和内存管理单元（Memory Management Unit）实现。通过MMU将虚拟地址翻译成物理地址。

RISC-V中虚拟地址都是64bit，但是高25bit没有使用，供未来使用。剩下的39bit中，27bit用作index，12bit用作offset。因为一个页面的大小是4096字节，所以对应的offset为12bit。

物理内存地址是56bit，其中44bit是物理页面编号（PPN， Physical Page Number），剩余的12bit继承自虚拟地址。

RISC-V的page table结构如下，分为了三级页表结构。在Directory中一个条目被称为PTE（Page Table Entry）。前9bit作为L2用于索引最高级page目录，一级9bit，因此可以索引512个entry；再根据L1、L0进行索引。

其实介绍的就是计算机组成原理中的多级索引，先通过L2索引到2^9个页面作为第一级索引，其中的每个页面可以根据L1再索引到2^9，同理L0也能再索引到2^9个页面。因此，总共可以索引到2^27个页面。比单级索引的好处在于，如果只需要一个页面，单级索引需要的页表大小为2^27个PTE，而三级索引只需要每级一个页面总共3*2^9即可。

在Figure 3.2中，低10位就是PTE的Flag，记录了一堆标记位：

  • Valid，为1则表示该PTE合法，可以用于地址翻译。
  • Readable和Writable。表明你是否可以读/写这个page。
  • Executable，是否可以从该page中执行指令。
  • User表明该page是否可以被用户空间进程访问。
  • 其他的不太重要，偶尔会出现，前五个最重要。

页表缓存（TLB）

TLB全称是Translation Lookaside Buffer。在三级页表中，当从内存加载或存储数据的时候，需要做三次内存查询。为了降低查询代价，几乎所有的处理器都提供了TLB作为PTE的缓存。

在第一次查询时，TLB会保存虚拟地址到物理地址的映射关系，下一次访问可以直接走TLB。操作系统也不知道TLB如何工作，但是如果切换了页表，操作系统会告诉处理器当前正在切换页表，处理器会清空TLB，因为TLB中缓存不再有用。

Kernel Page Table

在xv6中，下图左边对应的是内核的虚拟地址空间，右边的上半部分是物理内存或者说是DRAM，右边的下半部分对应的是I/O设备。

右半部分的设计由硬件设计者决定，操作系统启动，会从地址0x80000000开始运行。在完成从虚拟地址到物理地址翻译后，地址高于0x80000000会进入DRAM芯片，低于0x80000000会走向不同的IO设备。

地址空间的创建

为了实现lab3，要理解以下的部分代码的功能。

kernel/vm.c中，包括了操作地址空间和页表的xv6代码。其核心数据结构是pagetable_t，一个指向RISC-V根页表的指针；一个pagetable_t可以是内核页表，也可以是一个进程页表。 其中，最核心的函数是walk和mappages，前者为虚拟地址找到PTE，后者为新映射装载PTE。

vm.c中，以kvm开头的函数操作内核页表；以uvm开头的函数操作用户页表。copyout和copyin复制数据到用户虚拟地址或从用户虚拟地址复制数据。

启动时，main调用kvminit来创建内核的页表，这个调用发生在xv6是RISC-V分页之前，所以地址直接引用物理内存。 kvminit首先分配一个物理内存页面来保存根页表页。然后它调用kvmmap来装载内核需要的转换。转换包括内核的指令和数据、物理内存的上限是PHYSTOP，并包括实际上是设备的内存。

void kvminit()
{
  kernel_pagetable = (pagetable_t)kalloc();
  memset(kernel_pagetable, 0, PGSIZE);

  // uart registers
  kvmmap(UART0, UART0, PGSIZE, PTE_R | PTE_W);

  // virtio mmio disk interface，外设接口
  kvmmap(VIRTIO0, VIRTIO0, PGSIZE, PTE_R | PTE_W);

  // CLINT,对于外部中断处理
  kvmmap(CLINT, CLINT, 0x10000, PTE_R | PTE_W);

  // PLIC，对应内部中断处理
  kvmmap(PLIC, PLIC, 0x400000, PTE_R | PTE_W);

  // map kernel text executable and read-only.
  kvmmap(KERNBASE, KERNBASE, (uint64)etext - KERNBASE, PTE_R | PTE_X);

  // map kernel data and the physical RAM we'll make use of.
  kvmmap((uint64)etext, (uint64)etext, PHYSTOP - (uint64)etext, PTE_R | PTE_W);

  // map the trampoline for trap entry/exit to
  // the highest virtual address in the kernel.
  kvmmap(TRAMPOLINE, (uint64)trampoline, PGSIZE, PTE_R | PTE_X);
}

kvmmap是通过调用mappages实现装载的。mappages将虚拟地址到物理地址的映射装载到一个页表中，以页面大小为间隔，为每个虚拟地址单独执行此操作。对于需要映射的每个虚拟地址，mappages调用walk来查找该地址的PTE地址。然后初始化PTE，保存物理页号，所需权限（PTE_W,PTE_X和PTE_R）以及标记有效位的PTE_V。

walk的实现是在模拟RISC-V分页硬件。walk依次从三级页表中获得9bit。使用上一级的9bit虚拟位查找下一级或最终页面的PTE。如果PTE无效，则所需页面还没有分配；如果设置了alloc参数，walk就会分配一个新的页表页面，并将其物理地址放在PTE中。返回树中最低一级PTE地址。

main调用kvminithart来安装内核页表。它将根页表页的物理地址写入寄存器satp。之后，CPU将使用内核页表转换地址。

main调用procinit，为每一个进程分配一个内核栈。将每个栈映射到KSTACK生成虚拟地址。kvmmap将映射的PTE添加到内核页表中，kvminithart的系统调用将内核页表重新加载到satp中，以便satp知道新的PTE。

每个RISC-V的CPU将页表缓存在TLB上，当xv6更改页表时，必须告诉CPU使得相应的TLB无效。如果没有这么做，会导致一个进程可能会在其他进程的内存上修改（如果未被分配返回错误）。RISC-V中的指令sfence.vma用于刷新当前CPU的TLB。xv6在重新加载satp寄存器后，在kvminithart中执行sfence.vma，并在返回用户空间前，在用于切换到一个用户页表的trampoline代码中执行sfence.vma。

物理内存的分配

内核要在运行时，为页表、用户内存、内核栈和管道缓冲区分配和释放物理内存。xv6对内核末尾到PHYSTOP之间的物理内存进行分配。依次分配或释放4096字节的页面。使用链表将空闲页面记录下来，分配时将其从链表中删除。

分配器（allocator)位于kernel/kalloc.c中。allocator的数据结构是可供分配的物理内存页的空闲列表，每个空闲页的列表元素是一个struct run。使用自旋锁保护空闲列表（自旋锁适用于保持锁时间较短的情况）。

main函数通过调用kinit来初始化allocator。kinit初始化空闲列表以保存从内核末尾到PHYSTOP之间的每一页。xv6通过解析硬件配置信息确定有多少物理内存可用。kinit调用freerange将内存添加到空闲列表中，在freerange中每页都会调用kfree。

kfree函数首先将内存中的每一个字节设置为1，这就能使得无法读取到有效内容而结束程序，进而kfree使用头插法将其放入空闲列表。

kalloc删除并返回空闲列表中的第一个元素。

进程地址空间

每个进程都有一个单独的页表，当xv6在进程间切换时，页表也会切换。

如下图2.3所示，一个进程的用户内存从虚拟地址0开始，可用增找到MAXVA，原则上允许一个进程的内存寻址空间为256G。

当进程向xv6请求更多的用户内存时，xv6首先使用kalloc分配物理页面。然后，将PTE添加到进程页表中，指向新的物理页面。Xv6在这些PTE中设置PTE_W、PTE_X、PTE_R、PTE_U和PTE_V标志。大多数进程不使用整个用户地址空间；xv6在未使用的PTE中留空PTE_V。

页表的使用带来一下的便利：

  • 不同进程的页表将用户地址转换为物理内存的不同页面，这样每个进程都拥有自己的私有内存。
  • 每个进程看到的自己的内存空间都是以0地址起始的连续虚拟地址，而进程的物理内存可以是非连续的。
  • 内核在用户地址的顶部映射一个带有蹦床（trampoline）代码的页面，这样在所有地址空间都可以看到一个单独的物理内存页面。

图3.4更详细显示了xv6中执行态进程的用户内存布局。栈是单独的一个页面，显示由exec创建后的初始内容。包含命令行参数的字符串以及指向它们的指针数组位于栈的最顶部。再往下是允许程序在main处开始启动的值（即main的地址、argc、argv）。

通过guard page防止用户栈溢出（其PTE_V为0），试图访问则会返回页面故障，操作系统会自动为其分配更多内存。

汇编语言与RISC-V处理器

此处对应lec5中的内容，主要讨论的是汇编语言以及RISC-V的calling convention。

C程序到汇编

处理器并不能理解C语言，处理器能理解的是汇编语言，具体来说是二进制编码后的汇编语言。

对于一个RISC-V处理器，它能够理解RISC-V的指令集，因为它有一个关联的ISA(Instruction Sets Architecture)。ISA就是处理器能够理解的指令集，每一个指令都对应一个二进制编码或者Opcode。

C程序会通过处理器编译成ASM汇编文件再编译成二进制文件，即.obj或者.o文件。

在kernel.asm中可以看到汇编语言编译的内容。

RISC-V与x86

RISC-V的RISC指的是精简指令集（Reduced Instruction Set Computer），而x86通常被称为CISC，复杂指令集（Complex Instruction Set Computer）。

二者区别：

1、指令的数量，RISC更少。

2、RISC指令更简单。

RISC-V寄存器

上图就是RISC-V的寄存器的使用方式。

汇编语言是在寄存器上执行的，而非内存。因此经常见到的操作是用load将数据放在寄存器上，数据源可以来自内存也可以来自另一个寄存器。

使用寄存器的原因是运算和数据读取最快。a0到a7寄存器用来存储函数参数，如果一个函数有超过8个参数就需要使用内存。

对于上图中的第四列Saver列，简单的记忆方法：

  • Caller Saved寄存器在函数调用的时候不会保存
  • Callee Saved寄存器在函数调用的时候会保存

这里的意思是，一个Caller Saved寄存器可能被其他函数重写。假设我们在函数a中调用函数b，任何被函数a使用的并且是Caller Saved寄存器，调用函数b可能重写这些寄存器。我认为一个比较好的例子就是Return address寄存器（注，保存的是函数返回的地址），你可以看到ra寄存器是Caller Saved，这一点很重要，它导致了当函数a调用函数b的时侯，b会重写Return address。所以基本上来说，任何一个Caller Saved寄存器，作为调用方的函数要小心可能的数据可能的变化；任何一个Callee Saved寄存器，作为被调用方的函数要小心寄存器的值不会相应的变化。

stack

栈使得函数变得有组织，正常返回，以下是一个简单的栈的结构图，每一个区域都是一个Stack Frame，每一次调用都会产生一个Stack Frame。

对Stack来说，是从高地址到低地址使用，因此，栈是向下增长的。

有关Stack Frame有两个重要寄存器，第一个是SP（Stack Pointer），它指向Stack的底部，并代表了stack frame的位置。第二个是FP（Frame Pointer），它指向当前Stack Frame的顶部。因为Return address和指向前一个Stack Frame的的指针都在当前Stack Frame的固定位置，所以可以通过当前的FP寄存器寻址到这两个数据。

Isolation&system call entry/exit

trap机制

用户空间和内核空间切换的场景：

  • 执行系统调用
  • 出现page fault、运算除以0
  • 一个设备触发了中断，使得当前程序运行需要响应内核设备驱动

操作系统的一些high-level的目标能帮我们过滤一些实现选项。其中一个目标是安全和隔离，我们不想让用户代码介入到这里的user/kernel切换，否则有可能会破坏安全性。所以这意味着，trap中涉及到的硬件和内核机制不能依赖任何来自用户空间东西。比如说我们不能依赖32个用户寄存器，它们可能保存的是恶意的数据，所以，XV6的trap机制不会查看这些寄存器，而只是将它们保存起来。

在supervisor mode下，可以做两件事：

  • 读写控制寄存器。例如，读写SATP寄存器，也就是page table的指针；STVEC，也就是处理trap的内核指令地址；SEPC，保存当发生trap时的程序计数器；SSCRATCH等等。在supervisor mode你可以读写这些寄存器，而用户代码不能做这样的操作。
  • 它可以使用PTE_U标志位为0的PTE。当PTE_U标志位为1的时候，表明用户代码可以使用这个页表；如果这个标志位为0，则只有supervisor mode可以使用这个页表。

需要特别指出的是，supervisor mode中的代码并不能读写任意物理地址。在supervisor mode中，就像普通的用户代码一样，也需要通过page table来访问内存。如果一个虚拟地址并不在当前由SATP指向的page table中，又或者SATP指向的page table中PTE_U=1，那么supervisor mode不能使用那个地址。所以，即使我们在supervisor mode，我们还是受限于当前page table设置的虚拟地址

各种寄存器

  • SATP（Supervisor Address Translation and Protection）寄存器，它包含了指向page table的物理内存地址
  • STVEC（Supervisor Trap Vector Base Address Register）寄存器，它指向了内核中处理trap的指令的起始地址。
  • SEPC（Supervisor Exception Program Counter）寄存器，在trap的过程中保存程序计数器的值。
  • SSRATCH（Supervisor Scratch Register）寄存器，在trampoline 中使用，临时交换用。

trap代码执行流程

以write()为例：

  • 通过ecall指令进行系统调用，切换到内核态；
  • 执行uservec函数，位于trampoline.s文件中的汇编函数；
  • 跳转到C语言实现的usertrap中，位于trap.c中；
  • 执行syscall()函数，该函数会根据传入的数字进行查找，然后执行实现具体功能的函数。这里就是sys_write函数。
  • sys_write将显示的数据输出，然后返回给syscall()函数；
  • 在ecall之后，中断了用户代码的执行，接下来需要恢复用户空间执行。会第哦啊用一个usertrapret函数（trap.c）中，这个函数完成了部分方便在C代码中实现的返回到用户空间的工作。
  • 还有一些工作只能在汇编语言中完成。这部分工作通过汇编语言实现，存在于trampoline.s文件中的userret函数中。
  • 最终，在这个汇编函数中会调用机器指令返回到用户空间，并且恢复ECALL之后的用户程序的执行。

ECALL指令

老师在课上，通过gdb调试，一步步分析了该指令的功能，详细在笔记总结中可以看到。这里简要概述一下该指令的功能与特性。

在执行ecall前，查看页表可以发现，最后两条PTE的虚拟地址非常大，接近虚拟地址顶端，这两个page分别是trapframe page和trampoline page。这两个没有设置PTE_U的标志位，只能在内核态访问。

在执行ecall后，通过查看程序计数器（PC）可以发现，PC的值到了一个很大的地址上。通过PC可以发现，代码在trampoline页的最开始。

但是page table没有改变；寄存器也没有发生改变，仍然是用户寄存器。寄存器没有改变，则当前无法使用寄存器。

通过ecall走到trampoline page，而ecall实际上只会改变三件事情：

  • 将代码陷入内核态。
  • ecall将程序计数器的值保存在了SEPC（Supervisor Exception Program Counter）寄存器，在trap的过程中保存程序计数器的值。尽管其他的寄存器还是还是用户寄存器，但是这里的程序计数器明显已经不是用户代码的程序计数器。这里的程序计数器是从STVEC（Supervisor Trap Vector Base Address Register）寄存器拷贝过来的值。STVEC指向了内核中处理trap的指令的起始地址。
  • ecall会跳转到STVEC寄存器指向的指令。

因此，ecall做的工作离执行内核中的C代码还差的很远。接下来还需要做的事情：

  • 我们需要保存32个用户寄存器的内容，这样当我们想要恢复用户代码执行时，我们才能恢复这些寄存器的内容。
  • 因为现在我们还在user page table，我们需要切换到kernel page table。
  • 我们需要创建或者找到一个kernel stack，并将Stack Pointer寄存器的内容指向那个kernel stack。这样才能给C代码提供栈。
  • 我们还需要跳转到内核中C代码的某些合理的位置。

实际上，有的机器在执行系统调用时，会在硬件中完成所有这些工作。但是RISC-V并不会，RISC-V秉持了这样一个观点：ecall只完成尽量少必须要完成的工作，其他的工作都交给软件完成。这里的原因是，RISC-V设计者想要为软件和操作系统的程序员提供最大的灵活性，这样他们就能按照他们想要的方式开发操作系统。有以下的一些例子：

  • 因为这里的ecall是如此的简单，或许某些操作系统可以在不切换page table的前提下，执行部分系统调用。切换page table的代价比较高，如果ecall打包完成了这部分工作，那就不能对一些系统调用进行改进，使其不用在不必要的场景切换page table。
  • 某些操作系统同时将user和kernel的虚拟地址映射到一个page table中，这样在user和kernel之间切换时根本就不用切换page table。对于这样的操作系统来说，如果ecall切换了page table那将会是一种浪费，并且也减慢了程序的运行。
  • 或许在一些系统调用过程中，一些寄存器不用保存，而哪些寄存器需要保存，哪些不需要，取决于于软件，编程语言，和编译器。通过不保存所有的32个寄存器或许可以节省大量的程序运行时间，所以你不会想要ecall迫使你保存所有的寄存器。
  • 最后，对于某些简单的系统调用或许根本就不需要任何stack，所以对于一些非常关注性能的操作系统，ecall不会自动为你完成stack切换是极好的。

uservec:    
	#
        # trap.c sets stvec to point here, so
        # traps from user space start here,
        # in supervisor mode, but with a
        # user page table.
        #
        # sscratch points to where the process's p->trapframe is
        # mapped into user space, at TRAPFRAME.
        #
        
	# swap a0 and sscratch
        # so that a0 is TRAPFRAME
        csrrw a0, sscratch, a0

        # save the user registers in TRAPFRAME
        sd ra, 40(a0)
        sd sp, 48(a0)
        sd gp, 56(a0)
        sd tp, 64(a0)
        sd t0, 72(a0)
        sd t1, 80(a0)
        sd t2, 88(a0)
        sd s0, 96(a0)
        sd s1, 104(a0)

在该部分，首先执行csrrw指令，交换a0和sscratch两个寄存器内容。此时a0的值为0x3fffffe000，这是trapframe page的虚拟地址，而a0中的内容被保存到了sscratch寄存器中。在trampoline.S接下来的指令中就是将寄存器中的内容通过不同偏移保存在trapframe中。

然后，通过交换SATP（Supervisor Address Translation and Protection）和t1寄存器，将用户页表切换到内核页表。

**问题：**为什么代码没有崩溃？毕竟我们在内存中的某个位置执行代码，程序计数器保存的是虚拟地址，如果我们切换了page table，为什么同一个虚拟地址不会通过新的page table寻址走到一些无关的page中？

答案很简单，因为当前运行在trampoline代码中，之前提到trampoline代码在用户空间和内核空间都映射到同一个位置，是恒等映射。

Page Fault

此部分对应lec8内容。通过page fault机制可以实现一系列虚拟内存功能，包括：

  • lazy allocation
  • copy-on-write fork
  • demand paging
  • memory mapped files

几乎所有的操作系统都实现了以上功能，但是在xv6中，均未实现，一旦用户空间的进程触发page fault会导致进程被杀死。

到目前为止，内存地址的映射相对比较静态，page fault可以使得地址映射变得动态。通过page fault，内核可以更新page table。

当出现page fault，有三个信息非常重要：

  • 引起page fault的虚拟地址
  • 引起page fault的原因类型
  • 引起page fault的PC值，表明发生在用户空间的位置，便于恢复后返回

lazy page allocation

sbrk是xv6提供的系统调用，用于扩大自己的heap。当sbrk实际调用时，内核会分配一些物理内存，并将内存映射到用户应用程序的地址空间，并将内存内容初始化为0，再返回sbrk系统调用。同样也可以传入负数，减少内存地址空间。

xv6中默认的sbrk是eager allocation，一旦调用sbrk，内核会立即分配应用程式所需的物理内存。但是实际上，对于应用程序来说，很难预测自己需要多少内存，因此通常倾向于申请多余自己所需的内存，这意味着内存的消耗增加，并且部分内存永远不会被应用程序用到。

lazy allocation的核心思想是，在sbrk中，基本不做任何事，只将p->sz增加n（n为需要新分配的内存page数量）。但是内核并不会分配任何物理内存。在之后应用程序使用到新申请的内存会触发page fault。因为还没有将新的内存映射到page table，此时解析一个大于旧的p->sz，小于新的p->sz的虚拟地址，请求内核分配一个内存page，并重新执行指令。

Zero Fill On Demand

对一个用户的地址空间，会存在text区（程序的指令），data区（初始化的全局变量）和bss区（未初始化或初始化为0的全局或静态变量）。

在一个正常的操作系统中，如果执行了exec，会申请地址空间，里面会存放text和data。在bss中所有未初始化的全局变量都是0，而且会有许多page，可以调优的地方在于在物理内存中，只需要分配一个page，该page的内容全部为0，然后将虚拟地址空间全0的page映射到该物理page上，这样至少在程序启动时能节省大量物理内存的分配。

对于这个page，所有的PTE都是只读的。在修改的时候，会引起page fault，处理这个page fault需要在内存中申请一个新的page，内容置为0；然后更新page的映射关系，PTE可读可写；最后重新执行指令。

好处：

  • 类似lazy allocation，节省了空间
  • exec中需要做的工作变少了，程序可以更快地启动，更好的交互体验。

Copy On Write Fork

这是一个非常常见的优化，许多操作系统都实现了copy-on-write fork，也叫COW fork。

当Shell处理指令时，通过fork创建一个子进程。fork会创建一个进程的拷贝，子进程执行的第一件事就是调用exec运行一些其他程序，例如echo，这会丢掉之前的地址空间，使用包含echo的地址空间，会造成浪费。

在物理内存中，xv6中的shell 通常有4个page，调用fork时，基本就是创建了4个新的page，并将父进程中的内容拷贝到子进程中的4个page中。但是，调用了exec，又会释放这些page，分配新的page包含echo相关内容。

因此，对于这个特定场景存在一个非常有效的优化：当创建子进程时，使得子进程直接共享父进程的物理内存page，将子进程的PTE指向父进程对于的物理内存page。同时为了确保进程间的隔离性，将父子进程的PTE都设为只读。

当父或子进程需要修改内存内容，会得到page fault。这时，分配一个新的page，拷贝相应的物理page，并将新的PTE与之前的PTE都修改成可读可写。最后，重新执行用户指令（指调用userret函数）。

**Q：**当发生page fault时，我们其实是在向一个只读的地址执行写操作。内核如何能分辨现在是一个copy-on-write fork的场景，而不是应用程序在向一个正常的只读地址写数据？

**A：**对于PTE的标志位，之前介绍过第0bit到第7bit，但是没有介绍最后两位RSW。这两位保留给supervisor software使用，supervisor softeware指的就是内核。内核可以随意使用这两个bit位。所以可以做的一件事情就是，将bit8标识为当前是一个copy-on-write page。

Demand Paging

在未修改的xv6中，执行exec，操作系统会加载程序内存的text，data区域并以eager的方式加载进page table中。

但是根据先前的经验，考虑在实际使用的时候再从磁盘中将其加载到内存中。因此在exec中，为text和data分配好地址段，但是相应的PTE不对应任何物理page，将valid置为0。当实际使用的时候，加载进内存。

这里存在一个问题：内存耗尽该怎么办？

常见的选择是释放page，将部分page释放，重新分配，执行指令。

切换page的常用方式是LRU。同时对于dirty page 和non-dirty page通常选择切换non-dirty page。因为如果dirty page再被修改，则需要对它写两次（一次内存，一次文件）。

Memory Mapped Files

核心思想是将完整或者部分文件加载到内存中，这样就可以通过内存地址相关的load或store指令操纵文件。为了执行这个功能，操作系统会提供一个mmap的系统调用。它会接收一个虚拟地址（va），长度（len），protection，一些标志位，一个打开文件的文件描述符以及偏移量。

这里的语义就是，从文件描述符对应的文件的偏移量的位置开始，映射长度为len的内容到虚拟内存地址VA，同时我们需要加上一些保护，比如只读或者读写。

假设文件内容是读写并且内核实现mmap的方式是eager方式（不过大部分系统都不会这么做），内核会从文件的offset位置开始，将数据拷贝到内存，设置好PTE指向物理内存的位置。之后应用程序就可以使用load或者store指令来修改内存中对应的文件内容。当完成操作之后，会有一个对应的unmap系统调用，参数是虚拟地址（VA），长度（len）。来表明应用程序已经完成了对文件的操作，在unmap时间点，我们需要将dirty block写回到文件中。我们可以很容易的找到哪些block是dirty的，因为它们在PTE中的dirty bit为1。

当然，在任何聪明的内存管理机制中，所有的这些都是以lazy的方式实现。你不会立即将文件内容拷贝到内存中，而是先记录一下这个PTE属于这个文件描述符。相应的信息通常在VMA结构体中保存，VMA全称是Virtual Memory Area。例如对于这里的文件f，会有一个VMA，在VMA中我们会记录文件描述符，偏移量等等，这些信息用来表示对应的内存虚拟地址的实际内容在哪，这样当我们得到一个位于VMA地址范围的page fault时，内核可以从磁盘中读数据，并加载到内存中。所以这里回答之前一个问题，dirty bit是很重要的，因为在unmap中，你需要向文件回写dirty block。
