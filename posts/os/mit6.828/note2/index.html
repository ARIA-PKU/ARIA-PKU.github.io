lec10 多处理器与锁

为什么应用程序需要使用多个CPU核来提高性能？下面这张经典的图可以解释：

X轴为时间，Y轴为单位，具体意义取决于特定的曲线。核心在于，大概从2000年开始，CPU的时钟频率没有增加（绿线），这带来的结果是CPU的单线程性能到达极限（蓝线）；同时CPU的晶体管数量在持续增加（最上深红色线），因此现在为了使得运行更快无法通过提高单核性能，而要通过多核实现（黑线）。

如果系统调用并行在多个CPU核上，它们可能会并行访问内核中共享的数据结构，为了保证数据的一致性，需要使用锁来控制；同时，锁又反过来限制了性能。

critical section（临界区）：锁的acquire和release之间的代码，通常会以原子的方式执行共享数据的更新。

什么时候使用锁

一个保守同时又简单的规则：如果两个进程访问了一个共享的数据结构，并且其中一个进程会更新共享的数据结构，则需要对这个共享的数据结构加锁。

这个规则在有些时候过于严格，因为存在lock-free program，但是更为复杂；

矛盾的是，有时候这个规则过于宽松：除了共享数据外，其他场合也需要锁，例如printf，尽管这里没有共享数据，但是希望printf的输出是序列化的。

为每个数据结构自动加锁也会产生问题，比如将一个文件从一个目录d1移动到另一个目录d2。自动加锁则会先对d1加锁，删除文件，释放d1锁；再对d2加锁，增加文件，释放d2锁。

在完成第一步删除d1下的文件后，未执行第二步之前，其他进程会看到文件不存在，但是文件只是重命名，在任何一个时间点都要存在。因此，正确的解决方法是同时对d1和d2加锁，再删除添加文件，最后释放d1和d2的锁。

锁的特性和死锁

通常锁有三种作用：

  • 避免丢失更新。两个进程同时更新一个共享数据结构，可能会发生某个更新操作丢失。
  • 打包多个操作，使其具有原子性。critical section区域的使用。
  • 维护共享数据结构的不变性。只在锁内可修改。

死锁通常情况在于重复请求同一个锁，或者多个锁造成。

前者一般os可以检测并且panic；后者的解决方案是对锁进行排序，使得所有操作以相同顺序获得锁。

自旋锁（Spin lock）

自旋锁常见的实现方法是依赖于一个特殊的硬件指令，保证一次test-and-set操作的原子性。在RISC-V上，这个指令就是amoswap（atomic memory swap）。这个指令接收三个参数，分别是address，寄存器r1和r2。

amoswap(addr, r1, r2) {
	lock addr;		// 锁住地址
	tmp <- *addr;	// 将地址存放到临时变量（tmp）
	*addr <- r1;	// 将r1写入地址中
	r2 <- tmp;		// 将临时变量中的数据写入r2
	unlock addr;	// 将地址解锁
}

通过这里的加锁，可以确保address中的数据存放于r2，而r1中的数据存放于address中，并且这一系列的指令打包具备原子性。大多数的处理器都有这样的硬件指令，因为这是一个实现锁的方便的方式。这里我们通过将一个软件锁转变为硬件锁最终实现了原子性。不同处理器的具体实现可能会非常不一样，处理器的指令集通常像是一个说明文档，它不会有具体实现的细节，具体的实现依赖于内存系统是如何工作的。

// Acquire the lock.
// Loops (spins) until the lock is acquired.
void
acquire(struct spinlock *lk)
{
  push_off(); // disable interrupts to avoid deadlock.
  if(holding(lk))
    panic("acquire");

  // On RISC-V, sync_lock_test_and_set turns into an atomic swap:
  //   a5 = 1
  //   s1 = &lk->locked
  //   amoswap.w.aq a5, a5, (s1)
  // 这里就是循环查看锁
  while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ;

  // Tell the C compiler and the processor to not move loads or stores
  // past this point, to ensure that the critical section's memory
  // references happen strictly after the lock is acquired.
  // On RISC-V, this emits a fence instruction.
  __sync_synchronize();

  // Record info about lock acquisition for holding() and debugging.
  lk->cpu = mycpu();
}

// Release the lock.
void
release(struct spinlock *lk)
{
  if(!holding(lk))
    panic("release");

  lk->cpu = 0;

  // Tell the C compiler and the CPU to not move loads or stores
  // past this point, to ensure that all the stores in the critical
  // section are visible to other CPUs before the lock is released,
  // and that loads in the critical section occur strictly before
  // the lock is released.
  // On RISC-V, this emits a fence instruction.
  __sync_synchronize();

  // Release the lock, equivalent to lk->locked = 0.
  // This code doesn't use a C assignment, since the C standard
  // implies that an assignment might be implemented with
  // multiple store instructions.
  // On RISC-V, sync_lock_release turns into an atomic swap:
  //   s1 = &lk->locked
  //   amoswap.w zero, zero, (s1)
  __sync_lock_release(&lk->locked);

  pop_off();
}

Q：为什么release函数中不直接使用一个store指令将锁的locked字段写为0？

A：对于很多人包括我自己来说，经常会认为一个store指令是一个原子操作，但实际并不总是这样，这取决于具体的实现。例如，对于CPU内的缓存，每一个cache line的大小可能大于一个整数，那么store指令实际的过程将会是：首先会加载cache line，之后再更新cache line。所以对于store指令来说，里面包含了两个微指令。这样的话就有可能得到错误的结果。所以为了避免理解硬件实现的所有细节，例如整数操作不是原子的，或者向一个64bit的内存值写数据是不是原子的，我们直接使用一个RISC-V提供的确保原子性的指令来将locked字段写为0。

Q：在acquire函数的最开始，会先关闭中断。为什么会是这样呢？

A：spinlock需要处理两类并发，一类是不同CPU之间的并发，一类是相同CPU上中断和普通程序之间的并发。针对后一种情况，我们需要在acquire中关闭中断。中断会在release的结束位置再次打开，因为在这个位置才能再次安全的接收中断。

Q：编译器或者处理器可能会重排指令以获得更好的性能。这会改变指令流的正确性吗？

A：串行执行没有问题，但是对于并发执行会是一个灾难。如果我们将critical section与加锁解锁放在不同的CPU执行，将会得到完全错误的结果。所以指令重新排序在并发场景是错误的。为了禁止，或者说为了告诉编译器和硬件不要这样做，我们需要使用memory fence或者叫做synchronize指令，来确定指令的移动范围。对于synchronize指令，任何在它之前的load/store指令，都不能移动到它之后。锁的acquire和release函数都包含了synchronize指令。

lec11：线程切换

线程概述

线程的状态包含：

  • 程序计数器（PC），表示线程执行指令的位置
  • 保存变量的寄存器
  • 程序的栈，每个线程都有属于自己的栈，记录函数调用的内容，反映当前线程的执行点。

xv6的线程调度

对于线程切换，一些线程是自愿保存自己的状态并让其他线程运行。但是如果有一些程序正在执行一些长时间的计算任务，则无法自愿让出CPU给其他线程。

对于这种运算密集型线程，具体实现是利用定时器中断，将CPU控制权交给内核；位于内核的定时器中断处理程序，会自愿将CPU让出给线程调度器。

这样的处理流程称为pre-emptive scheduling（抢占调度），与之相对的是voluntary scheduling（自愿调度）。

有趣的是，在XV6和其他的操作系统中，线程调度是这么实现的：定时器中断会强制的将CPU控制权从用户进程给到内核，这里是pre-emptive scheduling，之后内核会代表用户进程（注，实际是内核中用户进程对应的内核线程会代表用户进程出让CPU），使用voluntary scheduling。

常见的线程状态：

  • RUNNING，线程当前正在某个CPU上运行（运行态）
  • RUNABLE，线程还没有在某个CPU上运行，但是一旦有空闲的CPU就可以运行（就绪态）
  • SLEEPING，这节课我们不会介绍，下节课会重点介绍，这个状态意味着线程在等待一些I/O事件，它只会在I/O事件发生了之后运行（等待态）

xv6的线程切换

从一个running的用户空间切换到另一个runable的用户空间的完整过程：

1、定时器中断强迫CPU从用户空间进程切换到内核，trampoline代码将用户寄存器保存到用户进程对应的trampframe对象中；

2、内核中运行usertrap，实际执行相应的中断处理程序。CPU开始在进程P1的内核线程和内核栈上执行内核中的C代码；

3、假设进程P1对应的内核线程决定让出CPU，则会做很多工作，后面再介绍，最后会调用swtch函数（switch是C中关键字，swtch用来避免冲突），这是线程切换的核心函数；

4、swtch函数会保存用户进程P1对应的内核线程寄存器在context对象。因此目前有两种寄存器：用户寄存器存放于trapframe中，内核线程寄存器存放在context中。

实际上，swtch函数不是直接从一个内核线程切换到另一个内核线程。xv6中，一个cpu上运行的内核线程可以直接切换到这个CPU对应的调度器线程。因此，如果运行在CPU0上，swtch函数会恢复之前为CPU0的调度器线程保存的寄存器和stack pointer，之后在调度器线程的context下执行scheduler函数。

在scheduler函数会做一些清理工作，例如将进程P1设为为runable状态；再通过进程表单找到下一个runable进程，假设是P2，scheduler函数会再次调用swtch函数，完成以下步骤：

1、保存自己的寄存器到调度器线程的context对象；

2、找到进程P2之前保存的context，恢复其中的寄存器；

3、P2调用swtch函数，返回到P2所在的系统调用或中断处理程序（注，因为P2进程之前调用swtch函数必然在系统调用或者中断处理程序中；

4、从用户空间进入内核空间会保存用户寄存器到trapframe。内核程序执行完成，trapframe用户寄存器就会恢复。

5、最后用户进程P2恢复运行。

每个CPU都有一个完全不同的调度器线程，调度器线程也是一种内核线程，拥有自己的context对象。任何运行在CPU1上的进程，当它决定出让CPU，它都会切换到CPU1对应的调度器线程，并由调度器线程切换到下一个进程。

Q：Linux系统上的多线程？

A：Linux是支持一个进程包含多个线程，Linux的实现比较复杂，或许最简单的解释方式是：几乎可以认为Linux中的每个线程都是一个完整的进程。Linux中，我们平常说一个进程中的多个线程，本质上是共享同一块内存的多个独立进程。所以Linux中一个进程的多个线程仍然是通过一个内存地址空间执行代码。如果你在一个进程创建了2个线程，那基本上是2个进程共享一个地址空间。之后，调度就与XV6是一致的，也就是针对每个进程进行调度。

Q：用户可以指定将线程绑定在某个CPU上吗？操作系统如何确保一个进程的多个线程不会运行在同一个CPU核上？要不然就违背了多线程的初衷了。

A：这里其实与XV6非常相似，假设有4个CPU核，Linux会找到4件事情运行在这4个核上。如果并没有太多正在运行的程序的话，或许会将一个进程的4个线程运行在4个核上。或者如果有100个用户登录在Athena机器上，内核会随机为每个CPU核找到一些事情做。
