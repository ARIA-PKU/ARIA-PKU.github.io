lec10 多处理器与锁

为什么应用程序需要使用多个CPU核来提高性能？下面这张经典的图可以解释：

X轴为时间，Y轴为单位，具体意义取决于特定的曲线。核心在于，大概从2000年开始，CPU的时钟频率没有增加（绿线），这带来的结果是CPU的单线程性能到达极限（蓝线）；同时CPU的晶体管数量在持续增加（最上深红色线），因此现在为了使得运行更快无法通过提高单核性能，而要通过多核实现（黑线）。

如果系统调用并行在多个CPU核上，它们可能会并行访问内核中共享的数据结构，为了保证数据的一致性，需要使用锁来控制；同时，锁又反过来限制了性能。

critical section（临界区）：锁的acquire和release之间的代码，通常会以原子的方式执行共享数据的更新。

什么时候使用锁

一个保守同时又简单的规则：如果两个进程访问了一个共享的数据结构，并且其中一个进程会更新共享的数据结构，则需要对这个共享的数据结构加锁。

这个规则在有些时候过于严格，因为存在lock-free program，但是更为复杂；

矛盾的是，有时候这个规则过于宽松：除了共享数据外，其他场合也需要锁，例如printf，尽管这里没有共享数据，但是希望printf的输出是序列化的。

为每个数据结构自动加锁也会产生问题，比如将一个文件从一个目录d1移动到另一个目录d2。自动加锁则会先对d1加锁，删除文件，释放d1锁；再对d2加锁，增加文件，释放d2锁。

在完成第一步删除d1下的文件后，未执行第二步之前，其他进程会看到文件不存在，但是文件只是重命名，在任何一个时间点都要存在。因此，正确的解决方法是同时对d1和d2加锁，再删除添加文件，最后释放d1和d2的锁。

锁的特性和死锁

通常锁有三种作用：

  • 避免丢失更新。两个进程同时更新一个共享数据结构，可能会发生某个更新操作丢失。
  • 打包多个操作，使其具有原子性。critical section区域的使用。
  • 维护共享数据结构的不变性。只在锁内可修改。

死锁通常情况在于重复请求同一个锁，或者多个锁造成。

前者一般os可以检测并且panic；后者的解决方案是对锁进行排序，使得所有操作以相同顺序获得锁。

自旋锁（Spin lock）

自旋锁常见的实现方法是依赖于一个特殊的硬件指令，保证一次test-and-set操作的原子性。在RISC-V上，这个指令就是amoswap（atomic memory swap）。这个指令接收三个参数，分别是address，寄存器r1和r2。

amoswap(addr, r1, r2) {
	lock addr;		// 锁住地址
	tmp <- *addr;	// 将地址存放到临时变量（tmp）
	*addr <- r1;	// 将r1写入地址中
	r2 <- tmp;		// 将临时变量中的数据写入r2
	unlock addr;	// 将地址解锁
}

通过这里的加锁，可以确保address中的数据存放于r2，而r1中的数据存放于address中，并且这一系列的指令打包具备原子性。大多数的处理器都有这样的硬件指令，因为这是一个实现锁的方便的方式。这里我们通过将一个软件锁转变为硬件锁最终实现了原子性。不同处理器的具体实现可能会非常不一样，处理器的指令集通常像是一个说明文档，它不会有具体实现的细节，具体的实现依赖于内存系统是如何工作的。

// Acquire the lock.
// Loops (spins) until the lock is acquired.
void
acquire(struct spinlock *lk)
{
  push_off(); // disable interrupts to avoid deadlock.
  if(holding(lk))
    panic("acquire");

  // On RISC-V, sync_lock_test_and_set turns into an atomic swap:
  //   a5 = 1
  //   s1 = &lk->locked
  //   amoswap.w.aq a5, a5, (s1)
  // 这里就是循环查看锁
  while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ;

  // Tell the C compiler and the processor to not move loads or stores
  // past this point, to ensure that the critical section's memory
  // references happen strictly after the lock is acquired.
  // On RISC-V, this emits a fence instruction.
  __sync_synchronize();

  // Record info about lock acquisition for holding() and debugging.
  lk->cpu = mycpu();
}

// Release the lock.
void
release(struct spinlock *lk)
{
  if(!holding(lk))
    panic("release");

  lk->cpu = 0;

  // Tell the C compiler and the CPU to not move loads or stores
  // past this point, to ensure that all the stores in the critical
  // section are visible to other CPUs before the lock is released,
  // and that loads in the critical section occur strictly before
  // the lock is released.
  // On RISC-V, this emits a fence instruction.
  __sync_synchronize();

  // Release the lock, equivalent to lk->locked = 0.
  // This code doesn't use a C assignment, since the C standard
  // implies that an assignment might be implemented with
  // multiple store instructions.
  // On RISC-V, sync_lock_release turns into an atomic swap:
  //   s1 = &lk->locked
  //   amoswap.w zero, zero, (s1)
  __sync_lock_release(&lk->locked);

  pop_off();
}

Q：为什么release函数中不直接使用一个store指令将锁的locked字段写为0？

A：对于很多人包括我自己来说，经常会认为一个store指令是一个原子操作，但实际并不总是这样，这取决于具体的实现。例如，对于CPU内的缓存，每一个cache line的大小可能大于一个整数，那么store指令实际的过程将会是：首先会加载cache line，之后再更新cache line。所以对于store指令来说，里面包含了两个微指令。这样的话就有可能得到错误的结果。所以为了避免理解硬件实现的所有细节，例如整数操作不是原子的，或者向一个64bit的内存值写数据是不是原子的，我们直接使用一个RISC-V提供的确保原子性的指令来将locked字段写为0。

Q：在acquire函数的最开始，会先关闭中断。为什么会是这样呢？

A：spinlock需要处理两类并发，一类是不同CPU之间的并发，一类是相同CPU上中断和普通程序之间的并发。针对后一种情况，我们需要在acquire中关闭中断。中断会在release的结束位置再次打开，因为在这个位置才能再次安全的接收中断。

Q：编译器或者处理器可能会重排指令以获得更好的性能。这会改变指令流的正确性吗？

A：串行执行没有问题，但是对于并发执行会是一个灾难。如果我们将critical section与加锁解锁放在不同的CPU执行，将会得到完全错误的结果。所以指令重新排序在并发场景是错误的。为了禁止，或者说为了告诉编译器和硬件不要这样做，我们需要使用memory fence或者叫做synchronize指令，来确定指令的移动范围。对于synchronize指令，任何在它之前的load/store指令，都不能移动到它之后。锁的acquire和release函数都包含了synchronize指令。

lec11：线程切换

线程概述

线程的状态包含：

  • 程序计数器（PC），表示线程执行指令的位置
  • 保存变量的寄存器
  • 程序的栈，每个线程都有属于自己的栈，记录函数调用的内容，反映当前线程的执行点。

xv6的线程调度

对于线程切换，一些线程是自愿保存自己的状态并让其他线程运行。但是如果有一些程序正在执行一些长时间的计算任务，则无法自愿让出CPU给其他线程。

对于这种运算密集型线程，具体实现是利用定时器中断，将CPU控制权交给内核；位于内核的定时器中断处理程序，会自愿将CPU让出给线程调度器。

这样的处理流程称为pre-emptive scheduling（抢占调度），与之相对的是voluntary scheduling（自愿调度）。

有趣的是，在XV6和其他的操作系统中，线程调度是这么实现的：定时器中断会强制的将CPU控制权从用户进程给到内核，这里是pre-emptive scheduling，之后内核会代表用户进程（注，实际是内核中用户进程对应的内核线程会代表用户进程出让CPU），使用voluntary scheduling。

常见的线程状态：

  • RUNNING，线程当前正在某个CPU上运行（运行态）
  • RUNABLE，线程还没有在某个CPU上运行，但是一旦有空闲的CPU就可以运行（就绪态）
  • SLEEPING，这节课我们不会介绍，下节课会重点介绍，这个状态意味着线程在等待一些I/O事件，它只会在I/O事件发生了之后运行（等待态）

xv6的线程切换

从一个running的用户空间切换到另一个runable的用户空间的完整过程：

1、定时器中断强迫CPU从用户空间进程切换到内核，trampoline代码将用户寄存器保存到用户进程对应的trampframe对象中；

2、内核中运行usertrap，实际执行相应的中断处理程序。CPU开始在进程P1的内核线程和内核栈上执行内核中的C代码；

3、假设进程P1对应的内核线程决定让出CPU，则会做很多工作，后面再介绍，最后会调用swtch函数（switch是C中关键字，swtch用来避免冲突），这是线程切换的核心函数；

4、swtch函数会保存用户进程P1对应的内核线程寄存器在context对象。因此目前有两种寄存器：用户寄存器存放于trapframe中，内核线程寄存器存放在context中。

实际上，swtch函数不是直接从一个内核线程切换到另一个内核线程。xv6中，一个cpu上运行的内核线程可以直接切换到这个CPU对应的调度器线程。因此，如果运行在CPU0上，swtch函数会恢复之前为CPU0的调度器线程保存的寄存器和stack pointer，之后在调度器线程的context下执行scheduler函数。

在scheduler函数会做一些清理工作，例如将进程P1设为为runable状态；再通过进程表单找到下一个runable进程，假设是P2，scheduler函数会再次调用swtch函数，完成以下步骤：

1、保存自己的寄存器到调度器线程的context对象；

2、找到进程P2之前保存的context，恢复其中的寄存器；

3、P2调用swtch函数，返回到P2所在的系统调用或中断处理程序（注，因为P2进程之前调用swtch函数必然在系统调用或者中断处理程序中；

4、从用户空间进入内核空间会保存用户寄存器到trapframe。内核程序执行完成，trapframe用户寄存器就会恢复。

5、最后用户进程P2恢复运行。

每个CPU都有一个完全不同的调度器线程，调度器线程也是一种内核线程，拥有自己的context对象。任何运行在CPU1上的进程，当它决定出让CPU，它都会切换到CPU1对应的调度器线程，并由调度器线程切换到下一个进程。

Q：Linux系统上的多线程？

A：Linux是支持一个进程包含多个线程，Linux的实现比较复杂，或许最简单的解释方式是：几乎可以认为Linux中的每个线程都是一个完整的进程。Linux中，我们平常说一个进程中的多个线程，本质上是共享同一块内存的多个独立进程。所以Linux中一个进程的多个线程仍然是通过一个内存地址空间执行代码。如果你在一个进程创建了2个线程，那基本上是2个进程共享一个地址空间。之后，调度就与XV6是一致的，也就是针对每个进程进行调度。

Q：用户可以指定将线程绑定在某个CPU上吗？操作系统如何确保一个进程的多个线程不会运行在同一个CPU核上？要不然就违背了多线程的初衷了。

A：这里其实与XV6非常相似，假设有4个CPU核，Linux会找到4件事情运行在这4个核上。如果并没有太多正在运行的程序的话，或许会将一个进程的4个线程运行在4个核上。或者如果有100个用户登录在Athena机器上，内核会随机为每个CPU核找到一些事情做。

lec13 睡眠和唤醒

XV6的线程切换中，不允许进程在执行switch函数的过程中，持有任何其他的锁。所以，进程在调用switch函数的过程中，必须要持有p->lock（注，也就是进程对应的proc结构体中的锁），但是同时又不能持有任何其他的锁。这也是包含了Sleep在内的很多设计的限制条件之一。

// Exit the current process.  Does not return.
// An exited process remains in the zombie state
// until its parent calls wait().
void
exit(int status)
{
  struct proc *p = myproc();

  if(p == initproc)
    panic("init exiting");

  // Close all open files.
  for(int fd = 0; fd < NOFILE; fd++){
    if(p->ofile[fd]){
      struct file *f = p->ofile[fd];
      fileclose(f);
      p->ofile[fd] = 0;
    }
  }

  begin_op();
  iput(p->cwd);
  end_op();
  p->cwd = 0;

  // we might re-parent a child to init. we can't be precise about
  // waking up init, since we can't acquire its lock once we've
  // acquired any other proc lock. so wake up init whether that's
  // necessary or not. init may miss this wakeup, but that seems
  // harmless.
  acquire(&initproc->lock);
  wakeup1(initproc);
  release(&initproc->lock);

  // grab a copy of p->parent, to ensure that we unlock the same
  // parent we locked. in case our parent gives us away to init while
  // we're waiting for the parent lock. we may then race with an
  // exiting parent, but the result will be a harmless spurious wakeup
  // to a dead or wrong process; proc structs are never re-allocated
  // as anything else.
  acquire(&p->lock);
  struct proc *original_parent = p->parent;
  release(&p->lock);
  
  // we need the parent's lock in order to wake it up from wait().
  // the parent-then-child rule says we have to lock it first.
  acquire(&original_parent->lock);

  acquire(&p->lock);

  // Give any children to init.
  reparent(p);

  // Parent might be sleeping in wait().
  wakeup1(original_parent);

  p->xstate = status;
  p->state = ZOMBIE;

  release(&original_parent->lock);

  // Jump into the scheduler, never to return.
  sched();
  panic("zombie exit");
}

从上面的代码看到，首先exit函数关闭了所有已经打开的文件。关闭文件系统中的文件涉及到引用计数，较为复杂；接下来，会将进程中对当前目录的记录释放。

对于进程的子进程，首先需要将子进程的父进程设置为init；所有进程退出的最后几步是由父进程的wait系统调用实现，将子进程的父进程设为init，可以由init调用wait完成最后几步。

然后，需要调用wakeup唤醒当前进程的父进程，父进程中的wait正在等待当前进程的返回，再将当前进程的状态设置为ZOMBIE。此时，进程还没有释放所有资源。最后通过sched进入调度器线程。

// Wait for a child process to exit and return its pid.
// Return -1 if this process has no children.
int
wait(uint64 addr)
{
  struct proc *np;
  int havekids, pid;
  struct proc *p = myproc();

  // hold p->lock for the whole time to avoid lost
  // wakeups from a child's exit().
  acquire(&p->lock);

  for(;;){
    // Scan through table looking for exited children.
    havekids = 0;
    for(np = proc; np < &proc[NPROC]; np++){
      // this code uses np->parent without holding np->lock.
      // acquiring the lock first would cause a deadlock,
      // since np might be an ancestor, and we already hold p->lock.
      if(np->parent == p){
        // np->parent can't change between the check and the acquire()
        // because only the parent changes it, and we're the parent.
        acquire(&np->lock);
        havekids = 1;
        if(np->state == ZOMBIE){
          // Found one.
          pid = np->pid;
          if(addr != 0 && copyout(p->pagetable, addr, (char *)&np->xstate,
                                  sizeof(np->xstate)) < 0) {
            release(&np->lock);
            release(&p->lock);
            return -1;
          }
          freeproc(np);
          release(&np->lock);
          release(&p->lock);
          return pid;
        }
        release(&np->lock);
      }
    }

    // No point waiting if we don't have any children.
    if(!havekids || p->killed){
      release(&p->lock);
      return -1;
    }
    
    // Wait for a child to exit.
    sleep(p, &p->lock);  //DOC: wait-sleep
  }
}

static void
freeproc(struct proc *p)
{
  if(p->trapframe)
    kfree((void*)p->trapframe);
  p->trapframe = 0;
  if(p->pagetable)
    proc_freepagetable(p->pagetable, p->sz);
  p->pagetable = 0;
  p->sz = 0;
  p->pid = 0;
  p->parent = 0;
  p->name[0] = 0;
  p->chan = 0;
  p->killed = 0;
  p->xstate = 0;
  p->state = UNUSED;
}

wait不仅是为了父进程方便直到子进程退出，实际上也是进程退出的一个重要组成部分。init进程的工作就是在一个循环不断调用wait，清理进程资源。

当父进程清理进程的所有资源，子进程的状态会被设置成UNUSED。之后，fork系统调用才能重用进程在进程表单的位置。

int
kill(int pid)
{
  struct proc *p;

  for(p = proc; p < &proc[NPROC]; p++){
    acquire(&p->lock);
    if(p->pid == pid){
      p->killed = 1;
      if(p->state == SLEEPING){
        // Wake process from sleep().
        p->state = RUNNABLE;
      }
      release(&p->lock);
      return 0;
    }
    release(&p->lock);
  }
  return -1;
}

如上代码，先扫描进程的表单，找到目标进程，将结构体的proc结构体中killed标志位设置为1，如果进程还在SLEEPING状态，将其设置为RUNNABLE。这里只是将killed标志位设置为1，并没有停止进程运行。

当目标进程运行到内核代码能安全停止运行的位置，检查killed标志位，如果设置为1，目标进程会自愿执行exit系统调用。在usertrap函数中可以看到安全退出位置。

所以kill系统调用并不是真正的立即停止进程的运行，它更像是这样：如果进程在用户空间，那么下一次它执行系统调用它就会退出，又或者目标进程正在执行用户代码，当时下一次定时器中断或者其他中断触发了，进程才会退出。所以从一个进程调用kill，到另一个进程真正退出，中间可能有很明显的延时。

对于SLEEPING状态的进程，如果它被kill了，它会被直接唤醒，包装了sleep的循环会检查进程的killed标志位，最后再调用exit。

lec14 文件系统

文件系统中核心的数据结构是inode和file descriptor。

inode代表一个文件的对象，不依赖于文件名，通过自身编号区分。file descriptor主要用于与用户进程交换，记录offset之类。

很多文件系统都有以下的分层结构：

  • 最底层的磁盘，提供持久化存储
  • buffer cache，比卖你频繁读写磁盘，将数据保存在内存中
  • logging层，为了保证持久性
  • inode cache，主要为了同步
  • inode，实现了write/read
  • 文件名和文件描述符操作

文件系统如何使用底层磁盘

比较常见的两种存储设备是SSD和HDD。SSD通常是0.1到1毫秒的访问时间，HDD通常在10毫秒量级完成读写一个disk block。

sector通常是磁盘驱动可以读写的最小单元，通常是512字节。block通常是操作系统或者文件系统视角的数据。它由文件系统定义，在XV6中它是1024字节。所以XV6中一个block对应两个sector。通常来说一个block对应了一个或者多个sector。

有的时候，人们也将磁盘上的sector称为block。所以这里的术语也不是很精确。

而文件系统的工作就是将所有的数据结构以一种能够在重启之后重新构建文件系统的方式，存放在磁盘上。虽然有不同的方式，但是XV6使用了一种非常简单，但是还挺常见的布局结构。

通常来说：

  • block0要么没有用，要么被用作boot sector来启动操作系统。
  • block1通常被称为super block，它描述了文件系统。它可能包含磁盘上有多少个block共同构成了文件系统这样的信息。我们之后会看到XV6在里面会存更多的信息，你可以通过block1构造出大部分的文件系统信息。
  • 在XV6中，log从block2开始，到block32结束。实际上log的大小可能不同，这里在super block中会定义log就是30个block。
  • 接下来在block32到block45之间，XV6存储了inode。我之前说过多个inode会打包存在一个block中，一个inode是64字节。
  • 之后是bitmap block，这是我们构建文件系统的默认方法，它只占据一个block。它记录了数据block是否空闲。
  • 之后就全是数据block了，数据block存储了文件的内容和目录的内容。

inode

一个inode是一个64字节的数据结构。通常包含以下内容：

  • type，表明inode是文件还是目录。
  • nlink，link计数器
  • size，文件数据包含多少字节
  • direct block，不同文件系统中的表达方式可能不一样，不过在XV6中接下来是一些block的编号，例如编号0，编号1，等等。XV6的inode中总共有12个block编号。这些被称为direct block number。这12个block编号指向了构成文件的前12个block。举个例子，如果文件只有2个字节，那么只会有一个block编号0，它包含的数字是磁盘上文件前2个字节的block的位置。
  • indirect block number，对应磁盘上的一个block，包含256个block。每个编号是4个字节，1024/4 = 256。

读取文件的第多少个字节可以通过偏移量求得。因此，inode中的信息足够用来实现read/write系统调用，至少可以找到对应的执行的block。

通过目录（directory）可以访问其他文件，对大部分Unix文件系统，一个目录本质上是一个文件加上一些文件系统能够理解的结构。在xv6中每个目录包含一个directory entries，每条entry有固定的格式：

  • 前2个字节包含目录中的文件或子目录的inode编号；
  • 后14个字节包含文件或子目录名

一个entry共16个字节。
